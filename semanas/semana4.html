Semana 4

Patrones Avanzados, Gestión de Datos Distribuidos y Observabilidad en Ecosistemas de Microservicios

El Cambio de Paradigma

La arquitectura moderna evoluciona desde los sistemas monolíticos estructurados —donde todos los módulos conviven y se ejecutan dentro de un único proceso de memoria— hacia **ecosistemas fragmentados**. En este nuevo entorno distribuido, los componentes están completamente aislados y se ven obligados a colaborar de forma asíncrona a través de la red.

Descomposición Funcional

Cada microservicio se enfoca de manera estricta y exclusiva en un único dominio de negocio, un concepto conocido como **Bounded Context** (Contexto Delimitado). Esto asegura que la complejidad de un módulo no contamine a los demás, minimizando el radio de explosión ante posibles fallos de infraestructura.

Despliegue Independiente

Permite romper las dependencias organizacionales entre los equipos de desarrollo. Un equipo autónomo puede liberar actualizaciones o parches críticos para el servicio de cobros o facturación de forma aislada sin comprometer el catálogo general, acelerando drásticamente el *Time-to-Market*.

1. Monolíticos vs. Fragmentados (Microservicios)

La descomposición de un sistema define cómo se distribuyen sus componentes y responsabilidades operativas en la infraestructura física y lógica de una organización.

Sistemas Monolíticos

Un sistema monolítico unifica toda la lógica de negocio, las capas de acceso a datos y la interfaz de usuario en un único artefacto empaquetado y desplegable.

  • WordPress (CMS): La lógica de administración de blogs, los temas visuales, el ecosistema de plugins y el core del sistema corren sobre el mismo servidor e interactúan directamente con una única base de datos central.
  • Sistemas ERP Tradicionales: Herramientas locales empresariales que integran la contabilidad, el inventario físico y los recursos humanos en un solo ejecutable masivo.
  • E-commerce Iniciales: Aplicaciones donde el flujo del carrito de compras, el catálogo de productos y el pasarela de pagos conviven en el mismo código fuente compartido.

Sistemas Fragmentados (Microservicios)

Un sistema fragmentado divide la aplicación en servicios atómicos independientes que cooperan de forma remota mediante protocolos de red estructurados (HTTP REST, gRPC o sistemas distribuidos de mensajería).

  • Netflix: Un servicio altamente especializado maneja el motor de recomendaciones, otro las cuentas y perfiles de usuario, y otro el streaming adaptativo de video; cada uno escala de forma elástica e independiente.
  • Uber: Servicios aislados para el rastreo de geolocalización, cálculo dinámico de tarifas en tiempo real, procesamiento seguro de pagos y algoritmos de asignación de conductores.
  • Amazon: El carrito de compras opera como un servicio autónomo con su propia persistencia de datos, completamente separado del motor de búsqueda y catalogación de productos.
2. Gestión de Datos: Saga Pattern vs. 2PC (Two-Phase Commit)

Garantizar y mantener la consistencia de la información cuando cada microservicio gobierna su propia base de datos es uno de los mayores desafíos en sistemas distribuidos. Existen dos estrategias principales:

Característica Patrón SAGA (Asíncrono) 2-Phase Commit (2PC)
Consistencia Eventual: El estado global se sincroniza progresivamente en el ecosistema tras unos segundos. Fuerte / Atómica: Todos los nodos cambian de estado de forma inmediata y simultánea.
Bloqueo de Recursos No bloqueante (Optimista): Las bases de datos locales confirman datos de inmediato, liberando hilos de ejecución. Sí (Bloqueante): Retiene y congela los recursos locales hasta que el coordinador central autoriza el commit.
Escalabilidad Alta: Altamente tolerante debido a su naturaleza asíncrona basada en eventos. Baja: Tiende a convertirse en un cuello de botella masivo a medida que se añaden nuevos servicios.
Estrategia ante Fallos Transacción Compensatoria: Si un paso del flujo falla, se ejecutan peticiones inversas lógicas para revertir los datos del pasado. Rollback Automático: El coordinador detecta el fallo en la fase de preparación y ordena abortar a todos los nodos de inmediato.
3. Patrones Fundamentales de Datos Distribuidos

Database-per-service

Concepto Principal: Cada microservicio es el dueño absoluto y exclusivo de su propio almacenamiento de datos. Está estrictamente prohibido que un servicio externo consulte directamente las tablas o esquemas de otro; la comunicación de datos se realiza únicamente a través de APIs públicas. Esto rompe el acoplamiento oculto y permite la poliglotía (ej: usar MongoDB y PostgreSQL en el mismo sistema).

Caso Real: En aplicaciones de delivery, el Servicio de Cuentas guarda credenciales en PostgreSQL, mientras que el Servicio de Rastreo maneja las coordenadas de los vehículos sobre Redis en tiempo real.

API Composition

Concepto Principal: Mecanismo de consulta directo donde un componente denominado "Compositor de API" intercepta la petición del cliente, realiza múltiples llamadas HTTP/gRPC en paralelo a los microservicios involucrados y unifica las respuestas fragmentadas en una única entidad JSON estructurada en la capa de Gateway.

Caso Real: Al renderizar el Dashboard de un usuario, el Gateway del sistema consulta en paralelo al Servicio de Facturas y al Servicio de Preferencias para servir toda la vista en una única llamada de red externa.

CQRS (Command Query Responsibility Segregation)

Concepto Principal: Segregación explícita de las responsabilidades de lectura y escritura del sistema. La lógica de alteración de estado (Commands) se procesa en motores altamente transaccionales, mientras que las consultas (Queries) se ejecutan de manera aislada contra réplicas optimizadas o cachés de memoria para maximizar el rendimiento.

Caso Real: En una ticketera masiva de eventos, la reserva del asiento (comando) ingresa a una base SQL transaccional estructurada, mientras que la vista en vivo de asientos disponibles para los millones de usuarios se sirve velozmente desde un clúster de Elasticsearch.

Idempotencia

Concepto Principal: Atributo de diseño que garantiza que una operación del backend conserve exactamente el mismo estado del sistema sin importar cuántas veces se intente ejecutar la misma petición idéntica. En redes propensas a reintentos automáticos, previene duplicaciones catastróficas.

Caso Real: Al procesar un pago digital, el servidor intercepta un identificador único global (UUID). Si el usuario presiona repetidamente el botón por problemas de latencia en su red, las peticiones duplicadas se ignoran al validar el ID existente.

4. Pilares de la Observabilidad y Metas de Rendimiento

En sistemas distribuidos, el monitoreo tradicional es insuficiente. La **Observabilidad** es la capacidad técnica de inferir y comprender el estado de salud interno de un ecosistema complejo basándose exclusivamente en el análisis de sus salidas externas en tiempo real. Responde de forma sistemática a tres etapas críticas: **Detección** (¿Qué falla?), **Aislamiento** (¿Dónde está el cuello de botella?) y **Diagnóstico** (¿Por qué ocurre?).

Logs Centralizados

Registro textual estructurado de un evento discreto con marcas de tiempo explícitas. Permite reconstruir detalladamente la línea de tiempo de un suceso específico dentro de un contenedor aislado.

Métricas y Alertas

Valores numéricos agregados y contables que reflejan el comportamiento operativo del hardware y software a lo largo del tiempo (por ejemplo, porcentaje de consumo de CPU o tasas de error HTTP 5xx).

Trazas Distribuidas

Representan el mapa del viaje completo de una transacción a través de la malla de microservicios. Indican con precisión exacta cuántos milisegundos consumió cada llamada en red.

Disponibilidad Estricta (SLA)

Los acuerdos de nivel de servicio (SLA) modernos para arquitecturas distribuidas exigen un estándar mínimo del 99.9% de uptime operativo anual, minimizando las ventanas permitidas de inactividad técnica.

Medición mediante Latencia P99

En microservicios, el promedio matemático es engañoso. Evaluamos la calidad mediante el **Percentil 99 (P99)**, garantizando que incluso el 1% de los usuarios que experimentan las condiciones de red más lentas reciban respuestas en rangos estables sub-200ms.

Herramientas del Ecosistema Tecnológico

Prometheus: Motor especializado en la recolección, almacenamiento temporal y consulta de **Métricas** en tiempo real mediante series de tiempo, configurando alertas predictivas ante saturación de servicios.

Stack ELK / Kibana: Ecosistema robusto encargado de centralizar, indexar y analizar volúmenes masivos de **Logs** distribuidos, facilitando búsquedas textuales cruzadas.

Jaeger: Plataforma de código abierto optimizada para la visualización de **Trazas (Traces)** distribuidas, indispensable para diagnosticar latencias y cuellos de botella en cascada entre servicios.

Service Mesh (Malla de Servicios): Capa de infraestructura dedicada (como *Istio* o *Linkerd*) encargada de interceptar y gestionar la seguridad, políticas de reintentos, encriptación TLS mutua y el enrutamiento transparente del tráfico entre microservicios sin contaminar el código de la aplicación.
