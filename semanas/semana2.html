Semana 2

Documentación de Decisiones Arquitectónicas utilizando Architecture Decision Records (ADR)

Documentación de la Arquitectura de Software

La documentación arquitectónica es un pilar fundamental en el ciclo de vida del software. Su objetivo principal es registrar de manera estructurada las decisiones cruciales que guían la construcción del sistema, asegurando que los fundamentos conceptuales estén disponibles para cualquier actor actual o futuro.

Propósito de Documentar:

Permite mitigar la pérdida de contexto técnico (amnesia arquitectónica), facilita la comunicación efectiva entre las partes interesadas (stakeholders), sirve como guía obligatoria para el equipo de ingeniería y actúa como base sólida para análisis futuros, auditorías y evaluaciones de calidad del sistema.

Concepto de la Sesión

Un Architecture Decision Record (ADR) es un artefacto documental que captura una decisión de diseño de software de gran impacto, detallando el contexto técnico en el que se formuló, las justificaciones comerciales/tecnológicas que motivaron su adopción y el conjunto de ventajas o desventajas resultantes del compromiso adquirido.

Nota del Autor (Michael Nygard): El concepto fue popularizado para evitar que las justificaciones de las decisiones de diseño se disuelvan en correos, chats o actas de reuniones inaccesibles, centralizándolas directamente junto al código fuente.

Calidad en la Documentación Arquitectónica

Una documentación ineficaz puede ser tan perjudicial como la ausencia total de la misma. Las diapositivas del curso exponen criterios claros de diferenciación:

Anti-patrones Comunes

  • Obsolescencia Dinámica: Documentos desactualizados que describen cómo funcionaba el sistema hace seis meses, generando confusión extrema.
  • Exceso de Detalles Inútiles: Documentación sobredimensionada que nadie lee, enfocada en minucias de implementación en vez de decisiones estructurales.
  • Falta de Contexto Histórico: Registrar un rotundo "usaremos X tecnología" sin explicar bajo qué restricciones, problemas o necesidades se tomó esa decisión.

Características Ejemplares

  • Precisión y Concisión: Debe ser directa, legible y enfocada estrictamente en responder el "por qué" de las decisiones complejas.
  • Mantenerse Actualizada: Debe tratarse como un entregable vivo dentro del ciclo de desarrollo, sincronizado estrechamente con el código base.
  • Estructura Consistente: Utilizar plantillas unificadas que permitan a cualquier miembro del equipo localizar rápidamente la información requerida.
Anatomía Estándar de un ADR

De acuerdo con las directrices académicas impartidas, un registro formal de decisiones debe componerse de las siguientes secciones secuenciales obligatorias:

1. Título: Identificador numérico único acompañado de una frase corta y descriptiva (ej. ADR-001: Implementación de caché corporativa).
2. Fecha y Estado: Especifica el momento exacto del registro y su estado actual (Propuesto, Aceptado, Rechazado, Supercedido).
3. Contexto: Plantea de forma detallada la necesidad técnica o comercial, las fuerzas en juego, las restricciones y el problema a resolver.
4. Decisión: Declaración concreta de la alternativa seleccionada para resolver el escenario planteado.
5. Consecuencias: Detalle explícito de los compromisos (trade-offs). Incluye los impactos positivos (beneficios) y negativos (costos/riesgos con su mitigación).
Casos de Estudio Prácticos e Implementaciones Realizadas
Ejercicio 1: El Dilema de la Base de Datos para un Catálogo Global

Enunciado: Una plataforma de comercio electrónico internacional necesita almacenar su catálogo de productos. El equipo debate si usar una base de datos relacional tradicional (como PostgreSQL) o una base de datos NoSQL orientada a documentos (como MongoDB). El catálogo requiere alta flexibilidad porque los productos tienen atributos muy diferentes según la categoría (ropa tiene tallas, tecnología tiene voltajes, libros tienen autores). El objetivo principal es la velocidad de lectura y la flexibilidad del esquema.

Tipo de Decisión Decisión de Tecnología de Persistencia y Estructura de Datos.
Atributos de Calidad Rendimiento (Latencia de Lectura) y Mantenibilidad (Flexibilidad del modelo de datos).
ADR-003: Adopción de Base de Datos Documental para el Catálogo de Productos
Estado ACEPTADO Contexto y Planteamiento del Problema

El catálogo de productos actual maneja una alta variabilidad de atributos que cambian constantemente según la categoría comercial. Diseñar esto en un modelo relacional (SQL) exigiría patrones complejos como Entity-Attribute-Value (EAV) o tablas altamente normalizadas, lo que degrada severamente el rendimiento de las consultas de lectura y complica la mantenibilidad del código debido a los múltiples joins.

Decisión

En el contexto de la alta variabilidad de atributos en los productos, de cara a optimizar la velocidad de lectura en la tienda virtual, decidimos adoptar MongoDB como base de datos del catálogo en lugar de una base de datos relacional, para obtener un esquema dinámico (schema-less) y lecturas de alta velocidad en un solo documento.

Consecuencias
  • Positivo (Flexibilidad): Los desarrolladores pueden agregar nuevos tipos de productos y atributos al vuelo sin necesidad de ejecutar migraciones de base de datos (alter tables).
  • Positivo (Rendimiento): Toda la información de un producto se recupera en una sola consulta indexada, mejorando los tiempos de respuesta del frontend.
  • Falta de Transacciones Complejas: Pérdida de restricciones de integridad referencial nativas (llaves foráneas) a nivel de motor.
  • Mitigación del Riesgo: La consistencia de datos de catálogo no requiere transacciones ACID complejas (a diferencia de los pagos). Cualquier validación crítica de integridad de IDs de categorías o proveedores se gestionará mediante lógica de validación en la capa de la aplicación (Backend).
Ejercicio 2: Comunicación entre Microservicios en Tiempo Real

Enunciado: Un sistema de logística de transporte necesita notificar a múltiples servicios internos (Facturación, Notificaciones Push, Seguimiento de Rutas) cada vez que un repartidor entrega un paquete. Hacerlo mediante peticiones HTTP/REST directas está generando acoplamiento temporal: si el servicio de Facturación se cae, el flujo completo de entrega falla y la app del repartidor se congela.

Tipo de Decisión Decisión de Integración de Sistemas y Estilo de Comunicación.
Atributos de Calidad Disponibilidad y Acoplamiento (Desacoplamiento de componentes).
ADR-004: Arquitectura Guiada por Eventos (EDA) mediante Message Broker
Estado ACEPTADO Contexto y Planteamiento del Problema

La comunicación síncrona mediante HTTP/REST entre el servicio de entregas y los sistemas secundarios está afectando la disponibilidad del sistema global. Si un servicio secundario experimenta latencia o caídas, se produce un efecto dominó que bloquea la aplicación móvil del usuario final.

Decisión

En el contexto de la necesidad de desacoplar los procesos secundarios del flujo principal de entrega, de cara a garantizar la alta disponibilidad de la aplicación móvil, decidimos implementar una arquitectura asíncrona guiada por eventos utilizando Apache Kafka en lugar de llamadas HTTP directas, para obtener un procesamiento de tareas en segundo plano totalmente aislado.

Consecuencias
  • Positivo (Resiliencia): Si el servicio de Facturación experimenta una caída, los mensajes se acumulan de forma segura en el Broker y se procesan automáticamente cuando el servicio se recupere, sin afectar al repartidor.
  • Positivo (Escalabilidad): Añadir nuevos interesados en el evento (por ejemplo, un servicio de analítica de datos) no requiere modificar el código del servicio de entregas.
  • Consistencia Eventual: Los datos no se actualizan instantáneamente en todos los sistemas al mismo milisegundo.
  • Mitigación del Riesgo: Se diseñará la interfaz de usuario para manejar estados transicionales (ej: "Procesando actualización") y se garantizará la idempotencia en los servicios consumidores para evitar procesar el mismo evento de entrega dos veces.
Ejercicio 3: Elección de Framework Frontend para una Aplicación Empresarial

Enunciado: Una startup financiera (Fintech) va a desarrollar su nueva plataforma de administración interna para agentes de soporte y analistas de riesgo. El equipo debe elegir entre construir una Single Page Application (SPA) tradicional con React, o una aplicación orientada a Server-Side Rendering (SSR) con Next.js. El SEO (indexación en Google) no es un factor relevante porque la plataforma requiere inicio de sesión obligatorio y es de uso privado.

ADR-005: Desarrollo de Plataforma Interna como Single Page Application (SPA) con React
Estado ACEPTADO Contexto y Planteamiento del Problema

Se requiere construir un panel administrativo complejo con alta interactividad y manejo de estados en tiempo real. Al ser una herramienta privada, los beneficios de SEO y primer renderizado veloz que ofrece el Server-Side Rendering (SSR) no justifican la complejidad ni el costo de mantener servidores Node.js ejecutándose 24/7 para renderizar vistas.

Decisión

En el contexto de un panel administrativo interno sin requerimientos de SEO, de cara a minimizar los costos de infraestructura y simplificar el despliegue, decidimos utilizar React en modalidad SPA (Single Page Application) en lugar de un framework SSR como Next.js, para obtener archivos estáticos que se pueden distribuir económicamente a través de un CDN (como AWS S3 + CloudFront).

Consecuencias
  • Positivo (Costos y Despliegue): El costo de hosting se reduce prácticamente a cero y el despliegue se reduce a subir archivos HTML/JS estáticos a un almacenamiento en la nube.
  • Positivo (Experiencia de Usuario): Una vez cargada la aplicación, la navegación entre pestañas es instantánea y fluida para los agentes.
  • Tiempo de Carga Inicial: La primera descarga del sitio puede ser pesada si el bundle de JavaScript crece demasiado.
  • Mitigación del Riesgo: Se implementará Code Splitting (división de código) nativo de React mediante React.lazy() y enrutamiento dinámico, asegurando que el navegador solo descargue las pantallas a las que el usuario intenta acceder.
Ejercicio 4: Migración de Monolito a Microservicios

Enunciado: Una empresa multimedia con un sistema monolítico muy grande está experimentando problemas porque el módulo de "Procesamiento y Compresión de Videos" consume el 90% de la CPU del servidor, provocando que las funciones sencillas (como el inicio de sesión o la pasarela de pagos) fallen por falta de recursos de hardware. El equipo decide separar ese módulo específico.

Tipo de Decisión Decisión de Descomposición de Sistemas y Patrón de Despliegue.
Atributos de Calidad Aislamiento de Fallos y Eficiencia de Recursos (Escalabilidad Eficiente).
ADR-006: Extracción del Módulo de Procesamiento de Video a un Microservicio Independiente
Estado ACEPTADO Contexto y Planteamiento del Problema

El procesamiento de archivos multimedia es una tarea intensiva en CPU que actualmente compite por recursos con el tráfico web transaccional dentro del mismo monolito. Esto genera problemas críticos de disponibilidad en la plataforma cuando múltiples usuarios suben contenido en simultáneo.

Decisión

En el contexto de la degradación del rendimiento transaccional debido a tareas pesadas de hardware, de cara a garantizar la estabilidad de las operaciones críticas del negocio, decidimos extraer el módulo de procesamiento de videos del monolito y transformarlo en un microservicio aislado, para obtener un entorno donde este componente pueda escalar de forma independiente.

Consecuencias
  • Positivo (Aislamiento de Fallos): Si el procesador de videos se satura o colapsa por falta de memoria, los usuarios podrán seguir iniciando sesión y comprando con normalidad en la plataforma principal.
  • Positivo (Optimización de Costos): Se pueden asignar servidores con GPUs o CPUs optimizadas exclusivamente para este microservicio, manteniendo el resto del sistema en servidores estándar más económicos.
  • Sobrecarga Operativa: Incremento en la complejidad del despliegue (CI/CD) y la necesidad de monitorear un sistema distribuido adicional.
  • Mitigación del Riesgo: Se utilizarán contenedores Docker gestionados mediante un servicio auto-escalable (como AWS ECS o Kubernetes) y se centralizarán los logs en un stack OpenTelemetry para mantener la observabilidad completa del flujo del video.

ADR-007: Desacoplamiento de Archivos Binarios mediante Almacenamiento de Objetos Externo
Estado ACEPTADO Contexto y Planteamiento del Problema

Guardar documentos pesados (PDFs/Imágenes) dentro de las tablas de la base de datos incrementa exponencialmente el tamaño de los archivos de datos (.mdf/.data). Esto ralentiza severamente los tiempos de indexación, degrada el rendimiento de la memoria caché del motor de base de datos y vuelve inviable la estrategia diaria de respaldos de seguridad debido al tamaño del archivo resultante.

Decisión

En el contexto de la necesidad de almacenar grandes volúmenes de documentos médicos estáticos, de cara a proteger el rendimiento y la mantenibilidad de la base de datos operacional, decidimos almacenar los archivos físicos en un servicio de almacenamiento de objetos externo (como AWS S3 o MinIO), guardando únicamente la URL de referencia (texto) dentro de la base de datos relacional.

Consecuencias
  • Positivo (Optimización de Base de Datos): La base de datos se mantiene ligera, rápida y sus respaldos periódicos se completan en pocos minutos en lugar de horas.
  • Positivo (Costo-Eficiencia): El costo por gigabyte en un almacenamiento de objetos (S3) es drásticamente inferior al costo de almacenamiento en discos de alto rendimiento para bases de datos (SSD aprovisionados).
  • Riesgo de Desincronización Órfana: Si un registro de la base de datos se borra pero el archivo en S3 no, o viceversa, se generan archivos "huérfanos" o referencias rotas.
  • Mitigación del Riesgo: Las eliminaciones de registros médicos nunca serán físicas (Hard Delete), se usará borrado lógico (Soft Delete con banderas is_deleted). Adicionalmente, se programará un script cron semanal que compare las URLs de la base de datos con los archivos del bucket para limpiar inconsistencias de manera automatizada.
Gestión del Impacto en el Equipo y Comunicación

Un error común es asumir que las decisiones arquitectónicas solo importan al equipo de backend. La realidad metodológica exige una gestión de comunicación adaptada a cada rol:

Rol del Equipo Impacto Primario de la Decisión Técnica Canal / Uso del ADR
Ingenieros / Devs Afecta directamente la selección de librerías, dependencias y las pautas de codificación en sus ramas de Git. Lectura del ADR antes de implementar funcionalidades asociadas al componente modificado.
Líderes Técnicos Afecta el cronograma, la estimación de tiempos (story points) y la complejidad operativa global. Revisión de viabilidad técnica y control de deuda técnica agregada al proyecto.
Especialistas de QA Modifica radicalmente los escenarios de prueba, la cobertura de automatización y las pruebas de carga. Diseño de planes de pruebas basados en los riesgos explícitos declarados en el ADR.
Filosofía "Documentation as Code" (Doc-as-Code)

Las últimas láminas del curso profundizan en cómo gestionar la configuración de estos artefactos. La industria moderna rechaza las plataformas externas aisladas (Word, Sharepoint) y promueve que los **ADRs coexistan con el código fuente** en formato de texto plano (**Markdown**).

Beneficios de Markdown

Permite versiones de texto plano súper ligeras, legibles tanto por humanos como por sistemas informáticos, y renderizables en cualquier plataforma de Git (GitHub, GitLab).

Control de Versiones (Git)

Los cambios en la arquitectura se aprueban mediante **Pull Requests (PR)**. Si una decisión cambia, no se borra el archivo; se crea un nuevo ADR y el anterior cambia su estado a **SUPERSEDED** (Superado).
