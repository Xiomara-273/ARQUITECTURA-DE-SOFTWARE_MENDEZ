Patrones de Diseño I

Creacionales y Estructurales: El Catálogo de Soluciones Reutilizables

Semana 6 | Arquitectura de Software | IST Yavirac

¿Qué son los Patrones de Diseño?

Introducidos formalmente por la Gang of Four (GoF), los patrones de diseño son soluciones estándar, probadas y optimizadas para resolver problemas comunes y recurrentes que surgen durante el desarrollo de software orientado a objetos[cite: 4, 5]. No se tratan de fragmentos de código directos para copiar y pegar, sino de plantillas conceptuales y guías arquitectónicas que estructuran las interacciones entre clases y objetos, evitando que los desarrolladores tengan que "reinventar la rueda" ante fallas de diseño complejas[cite: 7, 8].

La implementación de estas soluciones aporta dos ventajas críticas en ingeniería de software: un Lenguaje Común, que permite a los equipos de desarrollo comunicarse de manera clara y directa utilizando términos unificados (como decir "implementemos un Adapter") [cite: 6], y una alta capacidad de Reutilización, garantizando que el código final sea significativamente más flexible, modular, escalable y fácil de mantener frente a evoluciones o cambios en los requerimientos del sistema[cite: 7].

Categorías de la Gang of Four (GoF) [cite: 9]

1. Creacionales: Estos patrones se enfocan de manera exclusiva en los mecanismos de instanciación y creación de objetos dentro de un sistema[cite: 10, 13]. Su propósito central es aislar y encapsular el conocimiento sobre qué clases concretas está utilizando la aplicación en tiempo de ejecución, evitando por completo la dependencia rígida de asignaciones estáticas tradicionales mediante constructores directos.

Al abstraer el proceso de creación de instancias (como el uso repetitivo de la palabra reservada new)[cite: 14], el código base adquiere una flexibilidad superior. Esto faculta a los desarrolladores para decidir qué objetos específicos inicializar o reutilizar de forma dinámica de acuerdo con configuraciones cambiantes, entornos de despliegue o flujos específicos del frontend sin alterar los componentes que los consumen.

2. Estructurales: Esta categoría está dedicada por completo a la composición, organization e integración inteligente de múltiples clases, interfaces y objetos con el fin de estructurar arquitecturas de software de gran tamaño y alta complejidad[cite: 11, 36]. Su meta primordial es asegurar que los componentes del sistema colaboren entre sí eficientemente sin comprometer su modularidad individual ni romper la flexibilidad interna ante futuras modificaciones[cite: 37].

A través de estos patrones, los desarrolladores logran diseñar jerarquías limpias y robustas priorizando el principio de composición de objetos por encima de la herencia de clases tradicional. El resultado directo de este enfoque de diseño es un código altamente escalable donde interfaces incompatibles pueden traducirse fluidamente y los subsistemas densos se ocultan detrás de fachadas accesibles y simplificadas.

3. Comportamiento: Los patrones de comportamiento se centran específicamente en optimizar los algoritmos de comunicación, la distribución de responsabilidades y los flujos de interacción dinámica entre diversos objetos en tiempo de ejecución[cite: 12]. Su campo de acción va más allá de la mera interconexión estructural, analizando de manera profunda cómo se reparten las tareas lógicas complejas dentro de un flujo operativo.

Al estandarizar los canales y métodos de envío de mensajes cruzados entre componentes, estos patrones consiguen desvincular fuertemente a los objetos interdependientes. Esto da como resultado un software con un acoplamiento débil sumamente eficaz, donde las variaciones de comportamiento pueden añadirse de forma transparente (este bloque y sus implicaciones prácticas se abordarán a detalle en la Semana 7)[cite: 12].

1. Conceptos Fundamentales en Programación Orientada a Objetos

Para asimilar el comportamiento de los patrones creacionales y estructurales en el entorno de desarrollo, es indispensable dominar dos de las operaciones esenciales de la arquitectura de software:

¿Qué es Instanciar?

Instanciar es el proceso técnico de reservar memoria física en el sistema para dar vida a un objeto real y concreto a partir del plano conceptual definido por una clase. Mientras que la clase determina las propiedades y métodos teóricos, la instanciación ejecuta el constructor dinámicamente mediante la palabra reservada new en JavaScript, aislando un estado de datos único para ese elemento creado.

¿Qué es la Abstracción?

La abstracción es un pilar del diseño que consiste en aislar las características esenciales de un componente de software, ocultando las complejidades operativas e implementaciones internas que no añaden valor al consumidor del objeto. Permite definir interfaces públicas limpias, enfocándose firmemente en qué hace una entidad en lugar de cómo lo hace, reduciendo el acoplamiento cruzado.

2. Catálogo de Patrones Creacionales [cite: 13]

Categoría centrada en encapsular y abstraer los mecanismos de creación de objetos, evitando que el sistema dependa directamente de asignaciones rígidas o acoplamientos severos durante la instanciación[cite: 14].

Singleton

Restringe rigurosamente la instanciación de una clase para garantizar que exista únicamente una instancia viva en toda la aplicación, proporcionando un punto de acceso global y centralizado a su interfaz[cite: 16, 17].

Ejemplo 1: Gestor de Configuración Visual [cite: 27] Un controlador único cargado en la sesión del navegador para almacenar y sincronizar variables como las del tema gráfico (Dark o Light Mode).
Ejemplo 2: Carrito de Compras de la App Un objeto compartido globalmente a través de las diferentes vistas para evitar la pérdida o duplicidad accidental de los ítems seleccionados.

Factory Method

Define una interfaz común para la creación de objetos, delegando explícitamente en las subclases la responsabilidad de decidir qué clase concreta instanciar en tiempo de ejecución según el contexto[cite: 18, 19].

Ejemplo 1: Generador de Botones Dinámicos Una clase base de UI que delega si instanciar un BotonCierre o un BotonConfirmacion según los parámetros de un cuadro modal.
Ejemplo 2: Emisor de Alertas en Pantalla Un despachador que crea y añade al DOM de manera automatizada instancias de componentes tipo AlertaExito o bien AlertaError.

Abstract Factory

Proporciona una interfaz para crear familias completas de objetos vinculados o interdependientes entre sí, sin necesidad de acoplarse o especificar sus clases de implementación concreta[cite: 20, 21].

Ejemplo 1: Kits de Componentes UI (Temas) [cite: 63] Fábrica que produce en conjunto un BotonOscuro y un InputOscuro, impidiendo mezclas visuales inválidas en el frontend.
Ejemplo 2: Adaptadores de Entorno de Ejecución Generar un set completo de layouts adaptados si el usuario navega en un dispositivo móvil (MenuMobile) o web de escritorio.

Builder

Separa la construcción lógica de un objeto complejo de su representación final, permitiendo que un algoritmo secuencial paso a paso ensamble variaciones de un mismo elemento sin saturar constructores[cite: 22, 23].

Ejemplo 1: Constructor de Alertas Modales [cite: 63] Un script que expone encadenamiento de métodos como .setTitulo(), .conBotonCerrar() y .build() para armar popups personalizados.
Ejemplo 2: Generador de Formularios HTML Un objeto dinámico encargado de concatenar etiquetas de entradas como .agregarInput() and .agregarCheckbox() dinámicamente.

Prototype

Permite la duplicación exacta de objetos existentes clonando directamente su estado interno de datos guardados en memoria, evitando depender de llamadas costosas a constructores o instanciaciones pesadas desde cero[cite: 24, 25].

Ejemplo 1: Clonación de Tarjetas de Producto Clonar una estructura de tarjeta HTML (Product Card) que ya posee las clases de CSS inyectadas para reutilizar el nodo alterando solo el texto.
Ejemplo 2: Ajustes de Gráficos Estadísticos Duplicar un objeto de opciones complejas (ejes, colores, animaciones) para instanciar un segundo gráfico interactivo idéntico en pantalla.
3. Simulador de Patrón Creacional en Acción: Singleton [cite: 26]

El siguiente laboratorio interactivo emula el control rígido de un gestor de estilos visuales o recursos compartidos[cite: 27]. Si una instancia ya existe en el ecosistema, JavaScript intercepta la nueva llamada en el constructor y retorna la referencia previamente guardada en memoria[cite: 29, 32].

Controlador de Configuración de Tema de InterfazJavaScript (ES6)
class ThemeConfigManager {
    constructor() {
        if (ThemeConfigManager.instance) {
            console.warn("[ALERTA]: Instancia existente detectada. Reutilizando objeto compartido.");
            return ThemeConfigManager.instance;
        }

        console.log("[SISTEMA]: Inicializando gestor visual único por primera vez...");
        this.theme = "Dark Mode (Gris Ejecutivo)";
        ThemeConfigManager.instance = this;
    }

    obtenerConfiguracion() {
        return `Configuración Activa -> Tema: ${this.theme}`;
    }
}

const gestorA = new ThemeConfigManager();
const gestorB = new ThemeConfigManager();

console.log("¿Apuntan al mismo espacio de memoria?:", gestorA === gestorB);

4. Catálogo de Patrones Estructurales [cite: 36]

Se enfocan en la composición inteligente de clases y objetos, facilitando que entidades de gran tamaño colaboren de manera fluida mediante interfaces flexibles que evitan mutaciones destructivas en el código original[cite: 37, 38, 50].

Adapter

Convierte la interfaz incompatible de una clase en otra firma esperada por el cliente[cite: 39, 41]. Actúa como un traductor intermediario para que múltiples componentes se integren sin romper la lógica actual[cite: 42, 43].

Ejemplo 1: Conversor de Datos XML a JSON Un adaptador encargado de capturar una respuesta cruda en tags XML de un servidor viejo y mapearla a un objeto JSON plano consumible por el frontend.
Ejemplo 2: Unificación de Pasarelas de Pago [cite: 59] Encapsular métodos dispares de APIs externas (como PayPal o Stripe) bajo un método estandarizado local como .procesarCobro()[cite: 60].

Bridge

Desacopla por completo una abstracción funcional de su implementación técnica subyacente, permitiendo que ambas dimensiones escalen de forma totalmente independiente[cite: 44, 45].

Ejemplo 1: Interactividad de Reproductores Multimedia Separar la interfaz gráfica de botones de control de la lógica interna de APIs de video como YouTube, Vimeo o etiquetas nativas HTML5.
Ejemplo 2: Estructuras de Cuadros de Diálogo Aislar el componente abstracto VentanaModal de las implementaciones de renderizado particulares de hojas de estilo CSS.

Composite

Organiza y compone colecciones de objetos en estructuras arbóreas de jerarquía "parte-todo", forzando al sistema a tratar a objetos individuales y contenedores compuestos bajo una misma interfaz[cite: 46, 47, 48, 49].

Ejemplo 1: El Árbol del DOM HTML Estructura nativa donde un elemento simple (<img>) y un nodo contenedor compuesto (<div>) heredan la interfaz común de nodos web.
Ejemplo 2: Menús de Navegación Anidados Gestionar listas desplegables (Dropdowns) donde un enlace final es un ítem plano, pero una categoría es un conjunto que alberga más subenlaces.

Decorator

Adjunta de forma dinámica nuevas responsabilidades, estilos o comportamientos extras a un objeto individual en caliente, sustituyendo la herencia rígida por composición modular[cite: 51, 52].

Ejemplo 1: Inyección Dinámica de Clases CSS [cite: 63] Adicionar capas estéticas interactivas a un contenedor HTML básico (como .con-sombra o .gradiente-ejecutivo) según eventos de la UI.
Ejemplo 2: Validadores de Entradas de Formulario Decorar la función base de envío de un botón agregando capas sucesivas de sanitización y control de campos obligatorios vacíos.

Facade

Proporciona una interfaz unificada y drásticamente simplificada ante un subsistema de librerías o clases complejas, sirviendo como una fachada limpia que oculta la densidad técnica[cite: 53, 54].

Ejemplo 1: La API Nativa `fetch()` Actúa como fachada simplificada ocultando las complejas conexiones de red de bajo nivel y peticiones de las antiguas APIs de XMLHttpRequest.
Ejemplo 2: Orquestador de Animaciones Web [cite: 63] Crear una función única dispararIntro(elemento) que por dentro coordina opacidades, transformaciones y tiempos de retraso en milisegundos.

Flyweight

Optimiza y minimiza el uso de la memoria RAM de la aplicación compartiendo eficientemente datos e información común e invariable entre enormes colecciones de objetos similares[cite: 55, 56].

Ejemplo 1: Iconografía Compartida en Listas Reutilizar en memoria el mismo elemento vectorial de un icono institucional en una grilla con miles de registros escolares en lugar de duplicarlo por fila.
Ejemplo 2: Centralización de Hojas de Estilos Aplicar clases CSS globales para formatear textos corporativos en lugar de incrustar estilos pesados e individuales en línea (style="...").

Proxy

Establece un objeto sustituto o intermediario de control interceptando los accesos directos a un objeto real, permitiendo añadir validaciones de seguridad, almacenamiento en caché o cargas bajo demanda[cite: 57].

Ejemplo 1: Carga Perezosa de Imágenes (Lazy Loading) [cite: 57] Mostrar un marcador liviano o spinner temporal mientras la imagen pesada en alta resolución se descarga por completo en segundo plano.
Ejemplo 2: Caché Local de Respuestas HTTP [cite: 57] Interceptar consultas a las APIs escolares; si los datos se pidieron recientemente, el Proxy los extrae de inmediato de la memoria del localStorage.
5. Patrón Estructural en Acción: Adapter [cite: 58]

Modelado práctico de un escenario común en el frontend: el sistema universitario actual requiere renderizar listados de alumnos consumiendo un diccionario estructurado en JSON, pero el servidor antiguo devuelve las respuestas envueltas en cadenas XML. El Adaptador realiza la conversión sin alterar la fuente original.

Adaptador de Datos para la Interfaz EscolarJavaScript (ES6+)
// --- Clase Heredada (Legacy / Proveedor Externo Incompatible) ---
class CatatogoAcademicoXML {
    obtenerEstudiantesXML() {
        return "<estudiantes><estudiante><nombre>Karen Taquez</nombre><ciclo>6to</ciclo></estudiante></estudiantes>";
    }
}

// --- IMPLEMENTACIÓN DEL ADAPTER ---
class AdaptadorAcademico {
    constructor(sistemaXML) {
        this.sistemaXML = sistemaXML;
    }

    obtenerEstudiantesJSON() {
        const xmlCrudo = this.sistemaXML.obtenerEstudiantesXML();
        
        const nombreExtraido = xmlCrudo.match("<nombre>(.*?)</nombre>")[1];
        const cicloExtraido = xmlCrudo.match("<ciclo>(.*?)</ciclo>")[1];

        return {
            estudiante: {
                nombre: nombreExtraido,
                nivel: `${cicloExtraido} Semestre - Derecho`
            }
        };
    }
}

// --- Ejecución e Integración en la UI ---
const servicioIncompatible = new CatatogoAcademicoXML();
const adaptadorFront = new AdaptadorAcademico(servicioIncompatible);

const dataFormateada = adaptadorFront.obtenerEstudiantesJSON();
console.log("Objeto Adaptado Listo:", dataFormateada);

[ADAPTER]: Extrayendo literales y mapeando colecciones a objetos JSON...
Objeto Adaptado Listo: { estudiante: { nombre: 'Karen Taquez', nivel: '6to Semestre - Derecho' } }

7. Laboratorio Práctico de la Semana

Desafío integrador de desarrollo asignado para validar la correcta asimilación de las estructuras de acoplamiento débil estudiadas:

El Reto: Refactorizar un Catálogo Académico

Dada una clase vieja que exporta listas de estudiantes en formato XML estructurado, los alumnos deberán implementar un Adapter para entregar la misma data en formato JSON plano.

Objetivo: No alterar la clase original que genera el XML.

Criterios de Evaluación

  • Correcta separación de responsabilidades en la arquitectura.
  • Uso de tipos nativos de Python para realizar la conversión de datos de forma limpia.
  • Pruebas unitarias que demuestren que el cliente interactúa únicamente con la interfaz pública expuesta por el Adapter.
