Arquitecturas Reactivas y Evolutivas: Eventos (EDA), CQRS Extendido y Adaptabilidad Guiada

El Paradigma Reactivo

Los sistemas modernos no esperan pasivamente a que un usuario ejecute una acción síncrona; **reaccionan a cambios de estado en tiempo real**. La arquitectura orientada a eventos (EDA) rompe las cadenas del acoplamiento tradicional, permitiendo que múltiples sistemas colaboren de forma asíncrona mediante flujos continuos de datos e información.

1. Arquitectura Orientada a Eventos & Modelo Pub/Sub

En lugar de utilizar peticiones e interacciones directas entre servicios (las cuales congelan los hilos de ejecución), los componentes notifican cualquier acontecimiento o cambio de estado enviando un mensaje estructurado de forma asíncrona a través de un intermediario central.

Productores

Encargados de detectar los cambios de estado en el negocio y publicar el evento hacia la red sin preocuparse por quién lo va a escuchar.

Event Brokers

Canales o intermediarios de infraestructura (como **Apache Kafka** o **RabbitMQ**) que reciben, enrutan y garantizan la persistencia de los mensajes.

Consumidores

Servicios autónomos subscritos a canales específicos que se activan y reaccionan de manera independiente en cuanto llega un nuevo evento.

Ejercicio 1: Segregación de Responsabilidades con el Patrón CQRS

Estructuración del patrón que divide las acciones del sistema en **Comandos** (Escritura que altera el estado) y **Consultas** (Lectura pura desnormalizada). Esto evita bloqueos mutuos de procesos en la persistencia.

Estructura Lógica del Patrón CQRSTypeScript / JavaScript
// --- Capa de Escritura (Commands) ---
class RegistrarAsistenciaCommand {
    constructor(public readonly estudianteId: string, public readonly estado: string) {}
}

class RegistrarAsistenciaHandler {
    execute(command: RegistrarAsistenciaCommand) {
        // Ejecuta reglas de validación pesadas en el modelo transaccional
        console.log(`[WRITE MODEL]: Guardando registro inmutable del estudiante ${command.estudianteId} como [${command.estado}]...`);
    }
}

// --- Capa de Lectura (Queries) ---
class ObtenerRecordAsistenciaQuery {
    constructor(public readonly estudianteId: string) {}
}

class ObtenerRecordAsistenciaHandler {
    execute(query: ObtenerRecordAsistenciaQuery) {
        // Lee directamente desde un caché o vista desnormalizada veloz
        console.log(`[READ MODEL]: Recuperando DTO optimizado desde Redis para el ID: ${query.estudianteId}`);
        return { estudianteId: query.estudianteId, porcentajeAsistencia: "94%" };
    }
}

[READ MODEL]: Recuperando DTO optimizado desde Redis para el ID: EST-405

Ejercicio 2: Mecanismo de Arquitectura Orientada a Eventos (EDA)

Implementación de un Emisor de Eventos central (*Event EventEmitter*) que permite acoplar múltiples funciones observadoras independientes sin que el emisor conozca sus identidades internas.

Componente de Mensajería AsíncronaTypeScript
interface Observador {
    notificar(evento: string, datos: any): void;
}

class EmisorEventosAsistencia {
    private observadores: Observador[] = [];

    public suscribir(observador: Observador) {
        this.observadores.push(observador);
    }

    public disparar(evento: string, datos: any) {
        console.log(`[EVENT BROKER]: Canal emitiendo evento del negocio: -> "${evento}"`);
        this.observadores.forEach(obs => obs.notificar(evento, datos));
    }
}
Salida en Consola del Sistema
[EVENT BROKER]: Canal emitiendo evento del negocio: -> "AsistenciaMarcada"
[NOTIFICADOR]: Alerta instantánea SMS enviada al tutor legal.
[REPORTES]: Incrementando contador de alumnos en aula en tiempo real.

3. Consultas e Investigación Técnica Mandada a Hacer

Desarrollo analítico y conceptual de las preguntas académicas planteadas en clase sobre la auditoría técnica y persistencia avanzada en sistemas distribuidos.

A. ¿Qué es Event Sourcing?

Es un patrón arquitectónico de almacenamiento de datos donde, en lugar de guardar únicamente el valor o estado actual de un registro mediante sobreescrituras (CRUD tradicional), el sistema guarda de forma inmutable la **secuencia completa de eventos o cambios de estado** ocurridos a lo largo del tiempo. El estado actual se puede reconstruir en cualquier momento volviendo a procesar secuencialmente el historial de eventos desde el inicio.

B. ¿Qué es un Event Store?

Es una base de datos especializada de tipo *Append-Only* (solo inserción y lectura) optimizada exclusivamente para trabajar con Event Sourcing. No permite modificaciones (*Updates*) ni eliminaciones (*Deletes*) de registros existentes bajo ninguna circunstancia. Debe ser extremadamente rápida para añadir mensajes secuenciales de forma cronológica y proveer suscripciones eficientes a canales en tiempo real (Ejemplos: **EventStoreDB** o clústeres optimizados de **Apache Kafka**).

C. Diferencias entre CQRS y Event Sourcing

Criterio Patrón CQRS Event Sourcing
Enfoque Principal Separa la ruta de lectura de los datos de la ruta de escritura del sistema. Almacena la información como un historial secuencial e inmutable de eventos.
Base de Datos Puede trabajar sobre una única base tradicional SQL o NoSQL compartida. Requiere obligatoriamente un motor especializado de *Event Store*.
Relación Mutua Se puede implementar de forma aislada sin usar Event Sourcing. Casi siempre necesita acoplarse con CQRS para poder consultar los datos eficientemente.

D. Métricas Clave de la Arquitectura Evolutiva

Son los indicadores técnicos que las *Fitness Functions* analizan constantemente en el código para medir su capacidad de adaptación frente al cambio:

  • Acoplamiento Aferente (Ca): Cuántas clases externas dependen de un componente interno. Mide la estabilidad.
  • Acoplamiento Eferente (Ce): De cuántas clases externas depende un componente interno para funcionar. Mide el impacto ante cambios.
  • Complejidad Ciclomática: Número de rutas estructurales lineales independientes en el código. Evita el código espagueti.
  • Cobertura de Pruebas (Test Coverage): Nivel de respaldo automatizado que protege las reglas de negocio críticas de la degradación.
4. Arquitectura Evolutiva y Adaptabilidad

El software no es un ente estático. Una **Arquitectura Evolutiva** está diseñada específicamente para soportar el cambio constante, guiado e incremental a través de múltiples dimensiones a lo largo del tiempo, protegiendo las características esenciales del sistema contra la degradación técnica.

Fitness Functions (Funciones de Aptitud)

Son los guardianes matemáticos y automatizados del diseño de software. Se definen mediante métricas objetivas $$f(a) \rightarrow [0,1]$$ que evalúan de manera continua qué tan alineada está la estructura actual con sus objetivos clave de rendimiento y acoplamiento.

Implementación en CI/CD: Se integran de forma automática dentro de los flujos de compilación (como GitHub Actions) para detectar instantáneamente regresiones operativas antes de desplegar.
