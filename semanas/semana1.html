¿Qué es realmente la Arquitectura de Software?

"La arquitectura consiste en las decisiones de diseño estructural que son costosas y difíciles de modificar una vez implementadas."

La Programación (El Proceso)

Representa la colocación de los bloques de construcción y la mezcla de cemento. Se enfoca en resolver algoritmos específicos, la sintaxis correcta del lenguaje y asegurar la funcionalidad lógica inmediata del código fuente.

La Arquitectura (La Planificación)

Representa el diseño estructural macro. Determina la naturaleza funcional del edificio: si los requerimientos necesitan un Hospital (Alta Disponibilidad), una Discoteca (Alta Escalabilidad de carga masiva) o una Casa Habitacional (Optimización de costos estructurales).

El Gran Dilema: El "Trade-off" (Análisis de Sacrificios)

Un Arquitecto de Software no persigue una utopía informática perfecta; su misión es encontrar el equilibrio óptimo operativo. No existe un ecosistema tecnológico que sea 100% veloz, 100% impenetrable y 100% económico al mismo tiempo.

Axioma de Diseño Arquitectónico

"Para potenciar un atributo específico del sistema, invariablemente debemos sacrificar o mitigar otro."

Caso de Aplicación: Si se requiere que el sistema académico del IST Yavirac valide las notas usando Reconocimiento Biométrico Facial y Autenticación de Doble Factor (2FA):

  • ✅ Atributo Ganado: Máxima Seguridad, Confidencialidad e Integridad de las notas.
  • ❌ Atributo Sacrificado: Rendimiento de respuesta (Latencia), Usabilidad inmediata y un mayor Costo de infraestructura en la nube.
1. Modelado de Aplicaciones con UML (Caso de Estudio: Sistema Académico)

Concepto: El Lenguaje de Modelado Unificado (UML) es un estándar visual utilizado para especificar, visualizar y documentar los artefactos de un sistema de software. El uso de estereotipos gráficos e iconos visuales facilita la interpretación de los roles y dominios de las clases dentro de la arquitectura del sistema.

Ejemplo Real: En un sistema académico real, se implementa una superclase abstracta Persona que hereda los atributos comunes a Estudiante y Profesor. La relación fuerte de Composición define que si un registro de Estudiante es eliminado de la base de datos, su historial de Calificacion se destruirá en cascada inmediatamente para proteger la integridad referencial.

Atributos de Calidad Arquitectónicos (ASRs)

Los Architecturally Significant Requirements (ASRs) son requerimientos técnicos no funcionales pero determinantes que moldean de manera directa la infraestructura y patrones del software.

⏱️

Disponibilidad (Availability)

Porcentaje de tiempo operativo útil que el sistema responde de forma exitosa. Tradicionalmente se calcula con la métrica de los "nueves":

Nivel de NuevesTiempo Caído al Año
99% (Dos Nueves)3.65 días fuera de línea
99.9% (Tres Nueves)8.76 horas fuera de línea
99.999% (Cinco Nueves)5.26 minutos (Crítico)

Estrategia de Mitigación: Eliminar los Puntos Únicos de Fallo (SPOF) utilizando servidores redundantes y balanceadores de carga pasivos.

🚀

Escalabilidad (Scalability)

Capacidad del software de asimilar un incremento exponencial de la carga de transacciones o usuarios concurrentes sin degradar su servicio.

Vertical (Scaling Up)
Aumentar recursos físicos (RAM/CPU) al mismo nodo de servidor. Tiene límite físico y causa downtime.
Horizontal (Scaling Out)
Anexar múltiples nodos servidores económicos distribuidos mediante balanceo y microservicios. Infinito.
🥊

Resiliencia (Resilience)

Habilidad nativa del sistema informático para absorber fallos parciales o caídas de servicios de terceros y auto-recuperarse de manera autónoma.

"La metáfora del boxeador: La resiliencia no consiste en evadir los impactos, sino en poseer la capacidad de levantarse de la lona antes de la cuenta de diez."

Patrones de Implementación: Circuit Breaker (Disyuntor) y políticas de reintento automático (Retry loops).

🛡️

Fiabilidad (Reliability)

Probabilidad matemática de que el software ejecute sus funciones operativas sin fallos ni corrupción lógica bajo condiciones preestablecidas.

Distinción Fundamental de Conceptos:
• Disponibilidad: La puerta física del banco se encuentra abierta al público.
• Fiabilidad: El balance de mi cuenta corriente desplegado en pantalla es el matemáticamente correcto.
Durabilidad (Durability): "El Grabado en Piedra"

El principio de durabilidad se extiende de manera bidimensional en los entornos de producción moderna:

1. Durabilidad de Persistencia

Asegura que una vez confirmada una transacción (ej. subida de nota), los datos persistan de manera permanente en almacenamiento no volátil, volviéndose inmunes a cortes de energía o caídas del servidor físico.

2. Durabilidad del Ciclo de Vida

Determina qué tan bien envejece la arquitectura ante el paso del tiempo. Un código con alta durabilidad arquitectónica permite incorporar nuevas semanas académicas y dependencias sin viciar o colapsar el núcleo operacional antiguo.

2. La Tríada de Seguridad (CIA)

Todo arquitecto debe garantizar tres pilares fundamentales para proteger la información del sistema:

• Confidencialidad: Garantiza que la información solo sea accesible por personas autorizadas.

Ejemplo: Encriptar las contraseñas de los usuarios para que ni los administradores de la base de datos puedan leerlas.

• Integridad: Asegura que los datos no sean modificados de forma no autorizada o accidental.

Ejemplo: Usar firmas digitales en transacciones bancarias para asegurar que el monto no cambie durante el envío.

• Disponibilidad: Garantiza que los sistemas y datos estén listos para usarse cuando el usuario los necesite.

Ejemplo: Tener servidores de respaldo (replicación) para que, si uno falla, la aplicación siga funcionando sin que el usuario lo note.

3. Informe de Refactorización: Código Sucio vs. Clean Code

A continuación se detallan los 7 problemas críticos de diseño de software identificados en el Sistema Académico basado en Python y sus respectivas refactorizaciones aplicando principios de diseño de software.

01. Variables Globales → Encapsulamiento

Cuatro variables globales podían ser leídas y modificadas por cualquier función del archivo usando la instrucción global. Esto hace imposible razonar sobre el estado del sistema y dificulta escribir pruebas.

Antes (Código Sucio)
BD_ESTUDIANTES = [] BD_CALIFICACIONES = {} def procesar(...): global BD_ESTUDIANTES BD_ESTUDIANTES.append(...)
Después (Código Limpio)
class EstudianteRepo: def __init__(self, datos): self._db = {e.id: e for e in datos} def obtener(self, id): ...

02. Función Monolítica → SRP (Principio de Responsabilidad Única)

Una única función realizaba seis tareas completamente distintas: validar datos, agregar notas, calcular promedios, procesar cobros, escribir en disco y enviar correos. Modificar cualquier parte arriesgaba romper las demás.
