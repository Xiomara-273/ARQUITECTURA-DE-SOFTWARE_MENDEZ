Semana 3

Estilos Estructurales de Software, Estándares de Calidad ISO/IEC 25010 y Riesgos de Diseño

El Núcleo del Diseño Arquitectónico

El diseño de arquitectura no consiste en elegir tecnologías al azar, sino en encontrar un **equilibrio óptimo entre agilidad y estructura**. Cada decisión estructural responde a la necesidad de mitigar riesgos técnicos, organizar el código para el crecimiento del equipo y garantizar que el producto de software pueda mantener sus niveles de calidad a lo largo del tiempo.

¿Por qué adoptar un Estilo?

Un estilo de arquitectura provee un marco de trabajo abstracto que define cómo se estructuran las piezas de software, limitando las decisiones arbitrarias de los desarrolladores y permitiendo predecir el comportamiento del sistema bajo escenarios críticos de carga.

Factores de Selección

La elección de un estilo sobre otro (por ejemplo, Capas vs. Microkernel) se justifica mediante la evaluación rigurosa de restricciones organizacionales, el tamaño del equipo, las necesidades operativas de la infraestructura y el presupuesto disponible.

1. Estilos de Arquitectura de Software y Ejemplos de Aplicación

Los estilos arquitectónicos definen un patrón de organización estructural abstracta para los componentes de un sistema, estableciendo las reglas de comunicación y restricciones operativas entre ellos.

A. Arquitectura por Capas (Layered Architecture)

Aísla responsabilidades de manera secuencial (Presentación, Negocio, Datos). Cada nivel posee un rol restrictivo y cerrado por defecto, comunicándose únicamente con su capa inferior inmediata para maximizar el desacoplamiento.

5 Ejemplos de Software del Mundo Real:

1. Sistema Académico de Control de Notas (Yavirac) Python / Django
La vista HTML/JS captura la petición del alumno; la lógica en los controladores procesa si cumple las condiciones de asistencia, y el ORM gestiona de forma segura el registro físico en PostgreSQL.

2. Aplicación de Banca Web Comercial Java / Spring Boot
La interfaz bancaria (Capa Web UI) invoca servicios de transferencia que validan fondos y tokens (Capa de Negocio), antes de enviar la transacción al mainframe bancario (Capa de Datos).

3. Sistemas de Gestión de Contenido (WordPress) PHP
El núcleo maneja plantillas visuales (Temas), lógica estructurada de procesamiento de posts (Core) y consultas parametrizadas hacia bases de datos relacionales MySQL.

4. Plataformas de Facturación Electrónica .NET Core
La capa externa recibe los comprobantes XML, la capa intermedia aplica los esquemas de validación impositiva y firma digital legal, y la capa final almacena los históricos de auditoría.

5. Aplicaciones ERP Empresariales Básicas (Odoo) Python / JavaScript
Separa estrictamente la interfaz de administración del inventario, los módulos lógicos de cálculo de costos de manufactura, y los esquemas de almacenamiento de datos contables.

B. Arquitectura Microkernel (Plug-in Architecture)

Consiste en un núcleo centralizado (kernel) que ejecuta únicamente las operaciones mínimas esenciales para que el sistema funcione, permitiendo añadir extensiones operativas independientes (plugins) de forma dinámica sin apagar ni recompilar el sistema central.

4 Ejemplos de Software del Mundo Real:

1. Visual Studio Code / IDEs de Desarrollo Electron / TS
El kernel central provee únicamente el editor de texto básico y la gestión de archivos. Las funciones avanzadas (compiladores de Python, formateadores de Angular, linters) se inyectan como extensiones externas independientes.

2. Navegadores Web Modernos (Google Chrome) C++
El núcleo renderiza código HTML elemental y gestiona hilos de pestañas. Las extensiones descargadas de la Web Store añaden lógica complementaria (bloqueadores de anuncios, billeteras cripto) sin alterar el navegador base.

3. Reproductores Multimedia (VLC Media Player) C
El software central organiza las listas de reproducción y la ventana de video. Los codecs específicos de decodificación de audio y video (MP4, MKV, FLAC) funcionan como complementos de extensión acoplables.

4. Sistemas Operativos Monolíticos Flexibles (Linux Kernel) C
El núcleo primario maneja memoria y CPU de forma pura. Los controladores de hardware específicos (tarjetas gráficas ASUS TUF, teclados, adaptadores de red WiFi) se cargan dinámicamente como módulos.

C. Arquitectura de Pizarra (Blackboard Architecture)

Diseñada para problemas complejos donde no existe una estrategia algorítmica determinista conocida. Múltiples agentes de conocimiento independientes leen y escriben datos en un espacio de memoria compartido central (pizarra) para solucionar problemas de forma colaborativa.

7 Ejemplos de Software del Mundo Real:

1. Software de Reconocimiento de Voz Automatizado
Un agente analiza las frecuencias de audio aisladas, otro procesa los fonemas individuales, un tercer agente evalúa las reglas gramaticales y un orquestador final une todo en la pizarra para generar el texto final inteligible.

2. Sistemas de Inteligencia Artificial para Diagnósticos Médicos
Múltiples sub-algoritmos independientes especializados analizan de forma paralela la misma pizarra de datos clínicos del paciente (un agente revisa radiografías, otro el historial de laboratorio, otro los signos vitales) para consolidar un diagnóstico.

3. Sistemas de Control de Tráfico Aéreo
Agentes independientes recopilan constantemente datos distribuidos (radares de velocidad, sensores meteorológicos, rutas de vuelo programadas y niveles de combustible), publicándolos en una memoria compartida para calcular alertas de colisión.

4. Motores de Traducción de Lenguaje Natural Complejos
Divide el procesamiento textual en la pizarra a través de agentes dedicados a la traducción semántica estructural, validación de modismos culturales de la región y corrección ortográfica iterativa.

5. Compiladores Complejos y Avanzados de Optimización
El código fuente se almacena en la pizarra común mientras diferentes analizadores (léxico, sintáctico, semántico y optimizadores de registros de CPU) modifican progresivamente el árbol sintáctico abstracto sin depender directamente entre sí.

6. Sistemas Inteligentes de Conducción Autónoma (Vehículos Autónomos)
Los sensores LIDAR, las cámaras de video y los mapas GPS inyectan lecturas continuas en una pizarra central de la computadora interna. Agentes de frenado, aceleración y dirección leen esta información compartida en tiempo real.

7. Plataformas de Análisis Forense de Seguridad Informática
Herramientas distribuidas publican trazas de logs sospechosos en una base central. Agentes independientes evalúan patrones de malware, intentos de fuerza bruta e IPs bloqueadas para expresar una alerta de intrusión unificada.

2. Los 8 Pilares de Calidad (ISO/IEC 25010)

El estándar internacional **ISO/IEC 25010** sustituye a la antigua norma ISO 9126 y define los modelos integrales para la evaluación sistemática de los atributos cualitativos del software. Estos pilares guían al arquitecto para transformar requerimientos no funcionales en metas medibles.

Pilar de Calidad Descripción Técnica y Enfoque de Diseño
1. Adecuación Funcional Grado en que el software cumple con las necesidades explícitas e implícitas del negocio bajo condiciones específicas. Comprende la completitud, corrección y pertinencia funcional.
2. Eficiencia de Desempeño Relación entre el nivel de rendimiento del software y la cantidad de recursos de hardware consumidos bajo carga activa. Evalúa tiempos de respuesta, uso de recursos y capacidad del sistema.
3. Compatibilidad Capacidad de dos o más sistemas o componentes para intercambiar información y realizar funciones de forma cooperativa compartiendo el mismo entorno de hardware o software.
4. Usabilidad Facilidad con la que los usuarios finales pueden comprender, aprender y operar la interfaz de la aplicación de manera efectiva, atractiva y segura para sus objetivos.
5. Fiabilidad Capacidad del sistema para desempeñar sus funciones especificadas de manera correcta y continua durante periodos de tiempo definidos. Mide la tolerancia a fallos y la capacidad de recuperación.
6. Seguridad Protección de datos y sistemas ante accesos maliciosos malintencionados, garantizando la confidencialidad, integridad, no repudio y autenticidad de la información.
7. Mantenibilidad Efectividad y eficiencia con la que el software puede ser modificado por los desarrolladores para corregir errores, refactorizar, adaptarse a nuevos entornos o escalar. Incluye modularidad y reusabilidad.
8. Portabilidad Facilidad con la que el sistema de software puede ser transferido y ejecutado de un entorno de hardware, sistema operativo o nube a otra de forma nativa sin requerir adaptaciones masivas.
3. Riesgo de Diseño: Sinkhole Pattern

Concepto Avanzado: El patrón de sumidero o *Sinkhole Pattern* es un riesgo arquitectónico crítico que se manifiesta principalmente en las arquitecturas estructuradas por capas. Ocurre cuando una solicitud del usuario viaja secuencialmente a través de los diferentes niveles del sistema sin que se le agregue ningún valor técnico, validación, o lógica de negocio real en las capas intermedias, actuando como un simple puente pasamanos de datos.

El Impacto Operativo y la Regla Matemática

Este flujo innecesario consume ciclos de CPU, añade latencia en la comunicación interna y sobrecarga la memoria al crear múltiples instancias de objetos de datos idénticos (DTOs) que solo se reenvían de un punto a otro.

La Métrica de Control (Regla del 80-20): Un diseño saludable puede tolerar que un pequeño porcentaje de peticiones simples pasen directo. Sin embargo, si más del 20% de las solicitudes se comportan como un sumidero, la arquitectura pierde su justificación y se vuelve un problema de rendimiento.

Estrategia de Mitigación: Capas Abiertas (Open Layers)

Para solucionar el efecto Sinkhole, el arquitecto puede declarar ciertas capas operativas como **Abiertas (Open Layers)**. Esto rompe la restricción estricta de herencia secuencial, permitiendo que la capa de presentación o de control se salte la capa de servicios de negocio e invoque directamente a la capa de persistencia de datos únicamente en escenarios de consultas puras de lectura que no requieran procesamiento lógico.
